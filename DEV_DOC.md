# Developer Documentation

This document provides a technical overview of the Inception project. It serves as a guide for developers needing to deploy, modify, or debug the infrastructure. 

## 1. Environment Setup

### Prerequisites
Before setting up the environment, ensure the host machine meets the following requirements:
- A Linux-based environment (or WSL/VM equivalent) with a kernel supporting container namespaces.
- **Docker Engine** (or Docker Desktop) installed.
- **Docker Compose** plugin installed (v2+ recommended, executed via `docker compose`).
- **Make** build automation tool installed.

Additionally, due to the project's volume pathing requirements, the executing user must either have `sudo` privileges to automatically create the data directories, or the directories must be manually pre-created at `/home/${USER}/data/mariadb` and `/home/${USER}/data/wordpress`.

### Configuration Files (`.env`)
By default, the central configurations are managed through an environment file located at `srcs/.env`. This file is consumed by Docker Compose and injected as variables:
- `MYSQL_DATABASE`, `MYSQL_USER`, `MYSQL_HOST`
- `WP_URL`, `WP_TITLE`
- `WP_ADMIN_USER`, `WP_ADMIN_EMAIL`

Ensure `srcs/.env` is present before executing any build process. If you are developing locally, you must modify your `/etc/hosts` file manually to map the provided `WP_URL` to the host's localhost IP:
```text
127.0.0.1  samartin.42.fr
```

### Secrets
To adhere to container security best practices, sensitive data is explicitly decoupled from environment variables. 
Prior to building the project, ensure the following valid `.txt` files exist inside the `srcs/secrets/` directory:
- `db_password.txt` (WordPress DB User password)
- `db_root_password.txt` (MariaDB Global Root password)
- `wp_admin_password.txt` (WordPress application admin password)

These secrets are passed into the containers at runtime and mounted into `/run/secrets/` in memory.

## 2. Building and Launching the Infrastructure

The project structure is orchestrated using a single entrypoint `Makefile` at the repository root.

- **Initial Setup and Boot**:
  ```bash
  make
  ```
  This target inherently executes:
  1. Creation of the host's external data directories.
  2. `docker compose -p inception -f srcs/docker-compose.yml up -d --build`
  
  The `--build` flag ensures that any changes to your isolated `requirements/.../Dockerfile` files are cleanly compiled from scratch into their respective Alpine/Debian base images.

- **Restarting Active Containers**:
  ```bash
  make up
  ```
  Similar to the main rule, this brings the containers online detached, and guarantees any orphaned containers are stripped from the active Compose tree. 

## 3. Container and Volume Management Commands

Developers require fine-grained tools to manage and examine Docker objects generated by this repository. 

### Internal Make Targets
- **Container Shutdown**: 
  ```bash
  make down
  ```
  Stops the containers gracefully, retaining the networks and volumes via `docker compose down`.
  
- **Teardown & Volume Wiping**:
  ```bash
  make remove
  ```
  Tears down the `inception` infrastructure utilizing `docker compose down --rmi all --volumes`, and executes `sudo rm -rf` on the external data directories to simulate a clean slate.
  
- **Rebuilding Environment**:
  ```bash
  make re
  ```
  Executes the teardown, then initiates the build target again.

- **Developer Access**:
  To quickly shell into a specific service, run:
  `make access-nginx` | `make access-wordpress` | `make access-mariadb`

### Custom Docker CLI 
Should you need to deeply inspect objects outside the purview of the Makefile, use Docker's native CLI mappings:
- `docker ps -a` (Check container states and health-check outputs)
- `docker network inspect inception` (Evaluate internal bridge IP routing assignment)
- `docker volume ls` (View provisioned local volumes)
- `docker system prune -a --volumes` (DANGER: Completely prunes the host system of all unused Docker objects).

## 4. Persistent Project Data

Because containers are inherently ephemeral architectures, resetting or terminating a container destroys locally modified files.

To prevent dataloss across container lifecycles, Inception creates **Docker volumes configured as local bind mounts**.

**Where is the data stored?**
- **Database (`mariadb`):** Stored externally at `/home/${USER}/data/mariadb`. 
  Inside the container, this maps to `/var/lib/mysql`.
- **Web Content (`wordpress` & `nginx`):** Stored externally at `/home/${USER}/data/wordpress`. 
  Inside the containers, this maps to `/var/www/localhost/htdocs`.

**How does it persist?**
The `docker-compose.yml` configures these specific endpoints dynamically utilizing the generic driver:
```yaml
driver: local
driver_opts:
  type: none
  device: "/home/${USER}/data/..."
  o: bind
```
This forces Docker Native Volume behaviors strictly against the predefined host pathway. If a developer accidentally resets the `docker-compose` tree without calling `--volumes`, the database indices and live site configurations within those directories remain persistent and will immediately bind back onto the subsequent rebuild.
